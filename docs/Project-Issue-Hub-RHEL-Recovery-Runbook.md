# Project Issue Hub RHEL Recovery Runbook

This runbook describes how to rebuild a new RHEL server by combining:

- source code and schema from GitHub
- real business data from MySQL backup
- real attachment data from MinIO backup
- runtime settings from configuration backup

This is the correct recovery model for `Project Issue Hub`.

GitHub gives you:

- application code
- Docker and deployment files
- database schema and initialization SQL
- backup scripts and operations documents

GitHub does **not** give you:

- current production issue data
- current production users and assignments
- current production attachments
- current runtime secrets

Those items must be restored from backup.

---

## 1. Target Scenario

Use this document when:

- a new RHEL server is being prepared
- the original server is damaged or retired
- the full system must be rebuilt from backup

Assumptions:

- OS: `RHEL 8/9`
- deployment root: `/data/project-issue-hub`
- runtime: Docker Compose
- repository: `https://github.com/Keyiservice/Project-Issue-Hub.git`

---

## 2. Recovery Package Checklist

Before starting, confirm you have all four parts below.

### 2.1 GitHub repository

Required for:

- backend
- web frontend
- SQL initialization scripts
- Docker Compose files
- operational scripts

### 2.2 MySQL backup

At least one valid SQL dump:

- example path:
  - `/data/backups/project-issue-hub/mysql/opl_platform_2026-04-17_010000.sql`

### 2.3 MinIO backup

At least one valid object archive:

- example path:
  - `/data/backups/project-issue-hub/minio/minio_2026-04-17_010500.tar.gz`

### 2.4 Runtime configuration backup

At least one valid runtime config archive:

- example path:
  - `/data/backups/project-issue-hub/config/runtime_config_2026-04-17_011000.tar.gz`

The runtime config archive should contain:

- `.env`
- `docker-compose.yml`
- reverse-proxy config
- service unit

---

## 3. High-Level Recovery Order

Use this order strictly:

1. prepare the server
2. install Docker and Git
3. clone the repository from GitHub
4. restore runtime configuration
5. start infrastructure only: MySQL, Redis, MinIO
6. restore MySQL data
7. restore MinIO attachment data
8. start backend and web
9. restore Nginx and auto-start config if used
10. perform business validation

Do not start backend and web before MySQL and MinIO data are restored.

---

## 4. Prepare the New RHEL Server

Create required directories:

```bash
mkdir -p /data/project-issue-hub
mkdir -p /data/backups/project-issue-hub/mysql
mkdir -p /data/backups/project-issue-hub/minio
mkdir -p /data/backups/project-issue-hub/config
```

If backup files are stored elsewhere, copy them to the new server first.

Recommended verification:

```bash
ls -lh /data/backups/project-issue-hub/mysql
ls -lh /data/backups/project-issue-hub/minio
ls -lh /data/backups/project-issue-hub/config
```


## 7. Restore Runtime Configuration First

This step matters because your production configuration may differ from the
repository default. Do not assume GitHub alone contains the final production
runtime settings.

### 7.1 Extract the config backup

```bash
mkdir -p /tmp/pih-restore-config
tar -xzf /data/backups/project-issue-hub/config/runtime_config_YYYY-MM-DD_HHMMSS.tar.gz -C /tmp/pih-restore-config
find /tmp/pih-restore-config -maxdepth 3 -type f
```

### 7.2 Restore application runtime files

If the extracted archive contains the `project/` folder generated by the backup
script, restore files like this:

```bash
cp /tmp/pih-restore-config/project/.env /data/project-issue-hub/.env
cp /tmp/pih-restore-config/project/docker-compose.yml /data/project-issue-hub/docker-compose.yml
```

If present, also restore:

```bash
cp /tmp/pih-restore-config/project/nginx-project-issue-hub.conf /data/project-issue-hub/docs/ops/nginx-project-issue-hub.conf
cp /tmp/pih-restore-config/project/rhel-nginx-intranet.conf /data/project-issue-hub/docs/ops/rhel-nginx-intranet.conf
cp /tmp/pih-restore-config/project/project-issue-hub.service /data/project-issue-hub/docs/ops/project-issue-hub.service
```

### 7.3 Review effective runtime configuration

At minimum verify:

- ports
- MySQL runtime credentials
- Redis password
- MinIO endpoint and credentials
- JWT secret
- hostnames and reverse-proxy settings

Important:

- if your production deployment used a locally customized `docker-compose.yml`,
  restore that file before bringing containers up
- do not blindly start with repository defaults if production secrets were
  rotated

## 8. Start Infrastructure Only

Start only the stateful services first:

```bash
cd /data/project-issue-hub
docker compose up -d mysql redis minio
docker compose ps
```

Wait until MySQL is healthy:

```bash
docker compose ps
docker compose logs --tail=100 mysql
```

## 9. Restore MySQL Business Data

### 9.1 Confirm the target dump file

```bash
ls -lh /data/backups/project-issue-hub/mysql
```

Choose the SQL dump you want to restore.

### 9.2 Drop and recreate the target database

Use the root password or the approved recovery account from your restored
runtime configuration.

Example:

```bash
docker exec -i opl-mysql mysql -uroot -p'YOUR_MYSQL_ROOT_PASSWORD' -e "DROP DATABASE IF EXISTS opl_platform; CREATE DATABASE opl_platform CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
```

### 9.3 Import the SQL dump

```bash
docker exec -i opl-mysql mysql -uroot -p'YOUR_MYSQL_ROOT_PASSWORD' opl_platform < /data/backups/project-issue-hub/mysql/opl_platform_YYYY-MM-DD_HHMMSS.sql
```

### 9.4 Basic verification

```bash
docker exec -i opl-mysql mysql -uroot -p'YOUR_MYSQL_ROOT_PASSWORD' -e "USE opl_platform; SHOW TABLES;"
docker exec -i opl-mysql mysql -uroot -p'YOUR_MYSQL_ROOT_PASSWORD' -e "USE opl_platform; SELECT COUNT(*) AS issue_count FROM issue;"
docker exec -i opl-mysql mysql -uroot -p'YOUR_MYSQL_ROOT_PASSWORD' -e "USE opl_platform; SELECT COUNT(*) AS attachment_count FROM issue_attachment;"
```

If table names or row counts look obviously wrong, stop and do not continue.

## 10. Restore MinIO Attachment Data

Your current Compose setup uses a Docker named volume for MinIO. The simplest
recovery method is:

1. start MinIO once so the volume exists
2. stop MinIO
3. restore the archived `/data` content into that volume
4. restart MinIO

### 10.1 Stop MinIO before restore

```bash
cd /data/project-issue-hub
docker compose stop minio
```

### 10.2 Find the MinIO volume name

```bash
MINIO_VOLUME=$(docker inspect opl-minio --format '{{ range .Mounts }}{{ if eq .Destination "/data" }}{{ .Name }}{{ end }}{{ end }}')
echo "${MINIO_VOLUME}"
```

### 10.3 Restore archived MinIO data into the volume

```bash
docker run --rm \
  -v "${MINIO_VOLUME}:/restore-target" \
  -v /data/backups/project-issue-hub/minio:/backup \
  alpine:3.20 \
  sh -c "rm -rf /restore-target/* && mkdir -p /tmp/minio-restore && tar -xzf /backup/minio_YYYY-MM-DD_HHMMSS.tar.gz -C /tmp/minio-restore && cp -a /tmp/minio-restore/data/. /restore-target/"
```

If your production setup used a host-path MinIO data directory instead of a
named volume, restore the archive into that host path directly.

### 10.4 Restart MinIO

```bash
cd /data/project-issue-hub
docker compose up -d minio
docker compose logs --tail=100 minio
```

## 11. Start Backend and Web

Once MySQL and MinIO are restored:

```bash
cd /data/project-issue-hub
docker compose up -d backend web
docker compose ps
```

Check application logs:

```bash
docker compose logs --tail=200 backend
docker compose logs --tail=200 web
```

## 12. Restore Reverse Proxy and Auto-Start

If you use Nginx:

```bash
sudo cp /data/project-issue-hub/docs/ops/rhel-nginx-intranet.conf /etc/nginx/conf.d/project-issue-hub.conf
sudo nginx -t
sudo systemctl enable --now nginx
sudo systemctl restart nginx
```

If you use the provided systemd service:

```bash
sudo cp /data/project-issue-hub/docs/ops/project-issue-hub.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable project-issue-hub
sudo systemctl restart project-issue-hub
```

## 15. Common Failure Points

### 15.1 Wrong runtime secrets

Symptom:

- backend cannot connect to MySQL, Redis, or MinIO

Cause:

- repository defaults were used instead of restored production configuration

### 15.2 Database restored but attachments missing

Symptom:

- issue detail loads
- attachment preview fails

Cause:

- MinIO archive was not restored
- wrong MinIO volume was restored
- wrong `MINIO_PUBLIC_ENDPOINT` or access key was used

### 15.3 Attachments restored but not matched to records

Symptom:

- MinIO contains files
- issue detail still shows broken preview

Cause:

- MySQL dump and MinIO archive came from different backup points

Use backup sets from the same time window whenever possible.

### 15.4 GitHub clone works but business data is empty

This is expected if only GitHub was restored.

GitHub contains structure and code, not live business data.

---

## 16. Recommended Practice

For long-term maintainability, keep the following rule:

1. GitHub stores code, schema, scripts, and anonymous seed data
2. production MySQL data stays in backup storage, not GitHub
3. production MinIO attachments stay in backup storage, not GitHub
4. runtime secrets stay in protected configuration backup, not GitHub
5. perform a restore drill at least once per quarter

This separation is the correct enterprise operating model.
